# 📡 消息接入层可行性调研报告

> 调研人：mac龙虾 | 日期：2025-07-10

## 一、微信接入（⚠️ 最大坑位）

### 现状分析
| 方案 | 可行性 | 风险 | 说明 |
|------|--------|------|------|
| 微信公众平台（服务号/订阅号） | ✅ 官方支持 | 低 | 只能推送消息，不能做双向对话式工作流 |
| 企业微信 API | ✅ 官方支持 | 低 | 适合企业内部，有完善的 Webhook + 消息回调 |
| 微信个人号（第三方库） | ⚠️ 灰色地带 | **高** | itchat/wxpy 等已大面积封号，wechaty 企业版收费 |
| 微信开放平台（小程序） | ✅ 官方支持 | 低 | 可以做交互，但不是消息推送场景 |

### 结论
- **个人微信做消息接入 = 封号风险极高**，不建议作为 MVP 方案
- **企业微信**是可行替代方案，API 完善，适合工作流场景
- 如果必须支持个人微信，建议第一阶段用**微信公众号 + 关键词触发**做折中

## 二、Telegram 接入（✅ 最友好）

### API 能力
| 特性 | 支持情况 |
|------|---------|
| Bot API | ✅ 免费，官方文档完善 |
| Webhook 模式 | ✅ 支持，实时推送 |
| Long Polling | ✅ 备选方案 |
| 消息类型 | 文本/图片/文件/按钮/内联键盘 |
| 频率限制 | 约 30 msg/sec，对工作流足够 |
| Group/Channel | ✅ 支持群聊和频道 |

### 结论
- **Telegram 是 MVP 首选接入渠道**，零风险，API 最友好
- 一个 BotFather 创建 Bot 即可开始开发

## 三、邮件接入（✅ 稳定可靠）

### 方案对比
| 方案 | 说明 |
|------|------|
| IMAP 轮询 | 简单但延迟高，适合低频场景 |
| Gmail API (OAuth2) | ✅ 推荐，支持 Push Notification |
| QQ/163 SMTP+IMAP | ✅ 可用，已有 QQ 邮箱配置 |
| 专用邮箱监听服务 | 可部署独立服务监听 |

### 结论
- 邮件接入技术成熟，**MVP 阶段用 IMAP 轮询即可**
- 后续优化可接入 Gmail Push API

## 四、推荐 MVP 接入顺序

```
Phase 1 (Week 1-2):  Telegram + 邮件
Phase 2 (Week 3-4):  企业微信
Phase 3 (未来):      微信公众号（关键词触发）
```

## 六、风险总结

| 风险 | 等级 | 应对 |
|------|------|------|
| 微信个人号封号 | 🔴 高 | 放弃，用企业微信替代 |
| Telegram 国内访问 | 🟡 中 | 提供代理配置选项 |
| 邮件反垃圾 | 🟡 中 | 用白名单 + 验证机制 |
| 并发处理 | 🟢 低 | 消息队列异步处理 |

---

**结论：MVP 先做 Telegram + 邮件，微信用企业微信替代。技术上完全可行。**
