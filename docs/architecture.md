# FlowClaw MVP 架构方案

> 版本：v1.0 | 日期：2026-05-26 | 状态：已拍板

## 一、项目定位

跨平台 AI 工作流引擎 — 统一接入多渠道消息，通过 AI 智能路由与自动化处理，提供可视化工作流编排能力。

## 二、技术栈

| 层级 | 技术选型 | 理由 |
|------|----------|------|
| **运行时** | Node.js 18+ / TypeScript | 团队熟悉，AI 生态完善 |
| **包管理** | pnpm monorepo | 依赖隔离，workspace 共享 |
| **AI 框架** | LangChain.js | 支持复杂工作流编排，状态管理清晰 |
| **消息队列** | Redis + BullMQ | 异步处理，解耦消息接入与 AI 处理 |
| **数据库** | PostgreSQL | 关系型数据，MVP 阶段够用 |
| **缓存** | Redis | 会话状态、限流、热点数据 |
| **部署** | Docker Compose | MVP 阶段够用，后期可迁移 K8s |

## 三、核心架构

```
┌─────────────────────────────────────────────────────────────┐
│                     消息接入层 (Gateway)                      │
├─────────────┬──────────────┬──────────────┬─────────────────┤

└──────┬──────┴──────┬───────┴──────┬───────┴────────┬────────┘
       │             │              │                │
       ▼             ▼              ▼                ▼
┌─────────────────────────────────────────────────────────────┐
│                    统一消息网关                               │
│  • 消息标准化（统一格式）                                     │
│  • 渠道认证与限流                                            │

```

## 四、数据流

```
1. 用户发送消息到 Telegram Bot
2. Gateway 接收 → 转换为 UnifiedMessage → 推送到 BullMQ Queue
3. Engine Worker 消费消息 → AI 意图识别（分类：咨询/任务/告警等）
4. 根据意图匹配已配置的工作流 → 执行对应节点
5. 执行结果 → 通过 Gateway 回复用户
6. 记录执行日志 → 用于后续分析与优化
```
